## 1. 项目定位

### 1.0 背景与问题（Problem Statement）
车主在车内有大量碎片时间——通勤等红灯、充电桩前等待、接送家人的间隙、停车场等候。面前是一块车机大屏，但其商业化能力通常只有「加油 / 充电 / 导航」；想买点东西只能掏手机，副驾、后排同样如此。**车机这块屏在通用电商场景上是空白的。**
本项目要验证的核心问题：**在驻车场景下，能否以低干扰、安全可控的方式，在车机内完成「浏览 → 选购 → 下单 → 模拟支付 → 订单查看」的购物闭环，并清晰展示行车 / 断网 / 未登录下的安全门禁能力。** 本期为面向内部评审的概念 Demo（不绑定具体车厂、不接真实支付），用于把「车机商城体验长什么样、安全边界怎么守」讲清楚。

### 1.1 概述

- **是什么**：面向车机横屏场景的应用商城**系统**，含 **Android 车机端 + 后端服务 + Admin 管理后台**三部分。**并非单纯前端 Mock Demo**：第一版可用固定 Mock 账号、模拟支付、初始化商品数据，但商品/Banner/服务内容/订单/演示状态需经后台**管理与查看**，具备基础运营配置能力。
- **数据来源**：基础数据为**后端种子/初始化数据**（**不写死在前端**）；运行时支持 Admin 修改、前台实时读取。
- **目标**：向内部评审演示——驻车低干扰购物闭环（5–10 分钟跑完核心路径）；行车/断网安全门禁；实物/会员/展示服务三类差异化；车机端/后端/Admin 三端联动。
- **不是**：不接真实支付、不对外发布、不承接真实订单、不是技术框架展示。
- **应用标识**：显示名 `Apex Drive Store`；调试栏前缀 `DEMO`；主题仅**深色座舱（Dark Cockpit）**风格（无明暗切换）；语言仅简体中文。
- **演示账号**：`admin` / `123456`（前台 Mock 登录与 Admin 后台同账号）。

### 1.2 产品目标与成功假设

| 目标 | 产品假设 | 验证方式 | 成功判据 |
|------|----------|----------|----------|
| 验证车内购物可行性 | 驻车场景下，用户可在车机内理解商品、选择 SKU 并完成模拟交易 | A-01/A-02/B-01 端到端演示 | 核心路径 5–10 分钟内跑通，且演示者无需解释隐藏规则 |
| 验证安全边界 | 行车、断网、未登录三类状态能清晰阻断写入动作 | §3.3 门禁矩阵逐格验收 | 写入漏拦次数 = 0，冲突状态只出现最高优先级提示 |
| 验证商品类型差异 | 实物、会员、展示服务三类内容在 UI、流程、订单上有明确差异 | A-01/B-01/C-01 对比演示 | 评审可区分「可购实物」「可开通权益」「仅展示服务」 |
| 验证三端联动 | Admin 改内容后，前台和订单数据能实时反映 | Admin 商品/Banner/服务/订单/会话验收 | 改动无需重新构建前台即可生效 |

### 1.3 关键约束与依赖

- **评审对象**：内部产品、设计、工程、运营相关评审人员；不是终端消费者可用版本。
- **运行环境**：以 1280×720 横屏浏览器/Android WebView 为唯一目标环境；其余分辨率仅作非阻塞观察。
- **数据依赖**：种子数据由后端提供；前台实现阶段可用 MSW/Mock，但字段、枚举、文案必须与 §10/§11 一致。
- **安全依赖**：驾驶/网络/登录状态由 Demo Bar 手动模拟，不接真实车控、T-Box、网络探针或账号体系。
- **支付依赖**：支付为本地模拟，不调用支付机构、不展示真实收款码、不产生真实扣款。

## 2. 用户角色

| 角色 | 说明 |
|------|------|
| 驻车车主 | 主体用户，可完成实物购买与会员开通 |
| 行车驾驶员 | 仅低干扰浏览，所有写入动作被拦截 |
| 副驾乘客 | 可浏览商城内容，但任何交易动作仍需满足统一门禁（已登录+驻车+在线） |
| 演示者 | 通过 Demo Bar 控制登录/驾驶/网络状态与重置 |
| 后台运营 / 评审人员 | 通过 Admin 后台**管理**商品/Banner/服务内容/上下架，并查看订单、会员状态、演示会话 |

### 2.1 角色任务与关注点

| 角色 | 核心任务 | 关注点 | 本期体验边界 |
|------|----------|--------|--------------|
| 驻车车主 | 快速发现商品、完成购买或开通会员 | 操作路径短、价格清晰、支付结果可信 | 地址/车辆只读，不做真实售后 |
| 行车驾驶员 | 低干扰浏览，不进行交易写入 | 按钮状态明确、提示不打扰驾驶 | 行车时统一禁止输入与交易 |
| 副驾乘客 | 在车内协助浏览内容 | 能看清商品、服务与权益 | Demo 无法识别副驾，行车态仍统一拦截 |
| 演示者 | 控制状态、复现异常、重置演示 | 状态可见、切换即时生效、可快速回到初始态 | Demo Bar 是演示工具，不代表量产 UI |
| 后台运营 / 评审人员 | 管理内容、检查订单、观察会话 | 改动可追踪、前后台一致 | 不做真实履约、退款、售后 |

### 2.2 决策者与使用者

- **使用者**：车主/乘客在车机前台完成浏览与模拟交易；运营/评审在 Admin 后台完成内容与订单查看。
- **决策者**：内部评审人员根据演示闭环、安全门禁、三端联动与实现成本判断是否进入下一阶段。
- **维护者**：工程与设计团队基于本文、OpenSpec、测试矩阵维护行为一致性。

## 3. 全局状态与门禁规则

### 3.1 三组全局状态
| 维度 | 取值 | 默认 | 切换方式 |
|------|------|------|----------|
| 登录 | GUEST / LOGGED_IN | GUEST | 仅 Demo Bar 手动 |
| 驾驶 | PARKED / DRIVING | PARKED | 仅 Demo Bar 手动 |
| 网络 | ONLINE / OFFLINE | ONLINE | 仅 Demo Bar 手动 |

> 三组状态**均不自动触发**，只能由演示者手动操作 Demo Bar。

### 3.2 写入前置 & 优先级
- 所有写入动作（加购、改数量、结算、支付、开通会员）必须同时满足：`LOGGED_IN AND PARKED AND ONLINE`。
- 拦截优先级：`DRIVING > OFFLINE > GUEST`（多个限制同时存在时只显示最高优先级提示）。
- 登录态仅内存，应用重启回 GUEST。

### 3.3a 副驾身份识别（已知妥协）
Demo 阶段**无法识别当前操作者是驾驶员还是副驾/乘客**，因此 **DRIVING 时统一按行车态降级**——即使是副驾在操作，交易动作同样被拦。这是本期已知妥协；真实接入车厂后可结合座舱内传感器/座位识别再放开副驾。

### 3.3 门禁行为矩阵
| 操作 | GUEST+驻车+在线 | 已登录+驻车+在线 | 行车中 | 断网 |
|------|----------------|------------------|--------|------|
| 浏览商品/详情 | ✅ | ✅ | ✅ | ✅（已加载） |
| 加购/立即购买/开通会员 | LoginDialog | ✅ | 置灰+行车提示 | 置灰+断网提示 |
| 改购物车数量/删除/结算 | LoginDialog | ✅ | 置灰 | 置灰 |
| 模拟扫码成功 | LoginDialog | ✅ | 置灰 | 置灰 |
| 进入购物车/订单中心 | LoginDialog | ✅ | 可进入(只读) | 可进入(只读) |
| 查看"我的" | ✅ | ✅ | ✅ | ✅ |
| 搜索框输入 | ✅ | ✅ | 禁用(语音占位) | 禁用 |

## 5. 用户旅程地图与故事地图

### 5.0 一级目标
用户希望在车机内**快速浏览并购买商品，而不影响驾驶安全**。

### 5.1 User Map（总旅程，V1）
```
浏览商城
├── 进入首页：查看 Banner / 推荐商品 / 车主服务 / 搜索商品
├── 浏览商品：查看分类 / 商品列表 / 商品详情 / 查看 SKU
├── 购买商品：加入购物车 / 管理购物车 / 立即购买 / 确认订单 / 模拟支付
├── 查看订单：订单列表 / 订单详情
├── 开通会员：查看权益 / 确认车辆 / 模拟支付 / 查看会员状态
└── 浏览车主服务：充电服务 / 生活服务 / 后续能力介绍
```

### 5.2 角色旅程（线性）
- **驻车车主（购买）**：首页 → 推荐商品 → 商品详情 → 选 SKU → 加入购物车 → 购物车结算 → 确认订单 → 模拟支付 → 支付成功 → 查看订单。
- **会员用户（开通）**：首页 → 会员服务 → 查看权益 → 立即开通 → 确认车辆 → 模拟支付 → 会员生效 → 我的页查看状态。
- **行车用户（安全浏览）**：首页 → 浏览商品 → 查看详情 → 点击购买 → 系统拦截 → 提示「驻车后操作」。
- **后台运营（管理商品）**：后台登录 → 商品管理 → 新增商品 → 配置 SKU → 上传图片 → 上架 → 车机端展示。

### 5.3 故事地图三层
**第一层 · 用户活动**：浏览商品 → 选购商品 → 完成交易 → 管理订单 → 账号与车辆 → 演示控制 →（后台）内容运营

**第二层 · 用户任务**
- 浏览商品：查看首页推荐 / 按类目筛选 / 搜索商品
- 选购商品：实物详情+选 SKU / 会员详情 / 展示服务详情(不可购) / 加入购物车 / 立即购买 / 会员立即开通
- 完成交易：购物车结算 / 确认订单 / QR 扫码支付(模拟) / 查看支付结果
- 管理订单：查看订单中心 / 查看订单详情
- 账号与车辆：查看"我的"
- 演示控制：切登录 / 切行车驻车 / 切网络 / 重置数据
- 内容运营（Admin）：商品/Banner/服务 管理与上下架 / 订单查看 / 演示会话

**第三层 · Release 切片**
- **Release 1（核心闭环，P0）**：实物购物车购买(A-01) / 会员开通(B-01) / 展示服务拦截(C-01) / Demo 重置(DEMO-01)
- **Release 1.1（门禁与安全 + 后台，P1）**：未登录拦截+LoginDialog(AUTH-01) / 行车拦截(D-01) / 断网拦截(D-02) / Admin 管理后台 / 上下架
- **Release 1.2（体验完整，P2）**：实物立即购买(A-02) / 搜索(SEARCH-01) / 首页 / 类目 / 订单详情

### 12.1 错误码 → 前端行为
| 错误码 | 含义 | 前端行为 |

| 2002 | 断网拦截 | toast COPY-002 |
| 2003 | 范围超出(展示服务写入) | toast 错误消息或 COPY-039 |
| 4000 | 参数校验失败 | toast 错误消息 |
| 4004 | 资源不存在 | toast COPY-036 |

## 13. 不做范围（排除项）

第一版明确**不做**：真实支付、真实物流、真实库存扣减、退款/售后、优惠券、积分、发票、多账号注册、短信验证、地址编辑、车辆编辑、离线缓存、真实充电交易、地图核销、语音交易、应用内 AI 导购、支付失败/取消/超时、发货管理、售后管理；不对外发布、不承接真实订单；不做明暗主题切换、不做多语言、无性能压测。

**本期明确不做（讨论确认，补充）**：
- 会员到期 / 续费 / 退订（仅展示有效期数值）。
- 消息中心 / 收藏 / 地址管理（地址为固定只读，§10.4）。
- 搜索建议 / 历史 / 筛选 / 排序 / 联想词 / 类目内搜索（搜索仅商品名 substring，且仅实物+会员）。
- splash / 开屏引导页（冷启动直接进首页，§12.3）。
- 支付倒计时 / 恢复支付 / 支付凭证保存。
- **多分辨率自适应**：本期仅适配 **1280×720**，不做 1920×720/1080、2560×1440 等多分辨率自适应（其余分辨率不保证布局）。
- Admin 不做：发货 / 退款 / 售后 / 改价后重新支付 / 关闭订单。

**第一版重点验证**：车机商城浏览体验、驻车交易闭环、实物购买流程、会员开通流程、展示服务边界、行车/断网/未登录安全门禁、**后台商品与内容基础管理能力**、内部演示与评审可观察性。

### 13.1 本期实现范围与边界（Web-only）
> 技术栈与顺序见 [../project/decisions.md](../project/decisions.md) ADR-0004。

- **v1 系统范围 = 车机端(Web) + 后端服务 + Admin 管理后台**（三者皆属第一版，Admin 含 §9.5 六模块的真实管理能力，**非延后**）。
- **技术实现仅 Web**：React + TS + Vite + Tailwind，运行于横屏 PC 浏览器 1280×720 视口；车机以 Android WebView 承载同一构建产物。**不做原生 Android/iOS / Kotlin 端**（如后续追加，另立范围）。
- **数据为后端种子**（不写死前端）：后端 Express 提供统一 API，Admin 改、前台实时读；门禁 1001/2001/2002/2003 等校验在后端。
- **实现顺序（前端先行）**：先做车机端前台（可先用 MSW 对接，shape 同后端 §10 种子），随后补 Express 后端 + Admin。顺序是工程节奏，**不缩减 v1 范围**。
- 所有页面/文案/数据严格遵循本 PRD §9/§10/§11 与 [../design/design-tokens.md](../design/design-tokens.md)，不因实现方式改动 SSOT。
- **视觉以深色座舱（ADR-0007）为准**：本需求文档第十节「科技蓝/适度玻璃质感」为早期建议，已被深色座舱取代；其余原则（横屏/低干扰/大字大卡大按钮/路径短/左侧常驻导航/顶部调试可见/高对比）已并入设计原则。

#### 15.5.4 Admin 保存与联动规则

- **保存时机**：Admin 表单点击保存才提交；前台不感知未保存草稿。
- **校验顺序**：字段格式校验 → 引用目标存在性 → 类型/类目一致性 → 唯一性 → 保存。
- **前台刷新策略**：前台进入页面或返回列表时重新读取；已打开详情页可在下一次交互/刷新时感知改动，本期不强制 WebSocket 实时推送。
- **购物车价与状态**：购物车显示可保留加购时快照，但创建 checkout 必须后端重算当前价并校验 published/stock/SKU。
- **删除策略**：本期优先使用下架代替删除；若实现删除，删除不得破坏历史订单，且购物车含删除项结算应返回 4004/COPY-036。
- **排序策略**：所有前台列表先过滤 published，再按 sortOrder 升序，再按 productCode/bannerCode 稳定排序。

### 15.10 后端服务与接口行为细则

> 本节定义后端必须如何处理数据，而不只是有哪些 API。实现时可拆分 service/repository/controller，但行为必须一致。

#### 15.10.1 统一请求处理顺序

所有接口按以下顺序处理：

1. 解析请求路径、query、body。
2. 若接口需要登录，校验 token；失败返回 `1001`。
3. 若是写入接口，读取 `X-Demo-Drive` 与 `X-Demo-Net`，按 `DRIVING > OFFLINE > GUEST` 返回 `2001/2002/1001`。
4. 校验参数格式；失败返回 `4000`。
5. 校验资源存在；失败返回 `4004`。
6. 校验业务范围、published、stock、重复支付等业务冲突；失败返回 `2003/4009`。
7. 执行业务写入，返回 `{code:0,message:"OK",data}`。

#### 15.10.2 Catalog Service 细则

| 能力 | 输入 | 处理规则 | 输出 | 特殊情况 |
|------|------|----------|------|----------|
| 首页聚合 | 无 | banners 取 published=true；categories 固定 5 个(带 count)；featured 取 homeFeatured=true 且 published=true 按 sortOrder(change 0001：不截断)；shelves 按 5 类目分区返回全部 published 商品 | banners/categories/featured/shelves | 不补位；售罄仍返回并带 stock |
| 类目商品 | categoryCode | 校验类目存在；返回该类目 published=true 项，按 sortOrder/code 排序 | Product[] | 空数组合法 |

#### 15.10.4 Checkout Service 细则

| source/type | 输入 | 校验 | checkout items | 价格规则 | 成功后 |

#### 15.10.5 Payment/Order Service 细则

| 步骤 | 处理规则 |
|------|----------|

### 15.14 仍需进一步细化的点

以下内容如进入开发仍需在实现前最终确认：

- Admin 是否允许真正删除商品/Banner/服务；当前建议用下架代替删除。
- Product 图片/图标资源的实际来源与命名规则；当前只定义 assetKey，不定义素材库。
- Admin 表单是抽屉、弹窗还是独立页面；当前只定义行为，不锁 UI 形态。
- 搜索结果是否独立页面还是首页内结果区；当前只定义行为，不锁具体路由。
- 前台已打开详情页时 Admin 修改是否必须即时刷新；当前不要求 WebSocket，只要求下一次读取生效。

### 15.18 接口契约（请求/响应）

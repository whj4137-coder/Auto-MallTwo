# 车机商城调研与补充建议（Benchmarking）

> **这是什么**：对开源/商用车机商城与车载交互规范的调研，结合本项目需求，得出「需要补充」的清单。
> **来源说明**：本环境联网检索工具暂不可用，以下为基于公开车载设计规范（Android Automotive OS / Apple CarPlay HIG / 国内座舱产品）与产品常识的**综合归纳**，非实时抓取。标 ⚠️ 的结论建议联网恢复后二次核实。
> **产出**：补充项已落入 [../product/prd.md](../product/prd.md) §7（REQ-026 起）与 [../project/issues.md](../project/issues.md)。

## 2. 关键范式归纳（业界共识）

1. **行车锁定（Driving Lockout）**：行车时禁用文本输入与多步交易，仅保留浏览/语音。→ 与本需求「行车置灰 + 语音占位」一致。
2. **任务步数与单屏可视**：车载任务尽量 ≤ 少数步、单屏免滚动。→ 本需求已有「1280×664 无滚动」约束。
3. **大触控目标**：车载点击目标显著大于手机（常见 ≥ 64–76 dp）。→ 本需求未量化，建议补 token。
4. **手机接力 / 扫码支付**：车机少做敏感输入，支付转移到手机或扫码。→ 本需求「模拟扫码」即此范式。
5. **服务卡片化与"即将上线"占位**：未开放服务以展示卡 + 占位按钮呈现。→ 本需求展示服务即此。
6. **状态可见性**：当前车辆/账号/网络等状态常驻可见。→ 本需求 Demo Bar 已覆盖。
7. **降噪与一致反馈**：写入用瞬时 toast，不打断。→ 本需求已规定。

## 3. 与本需求对比：建议补充项

> 原则：本项目是「评审 Demo」，补充以**不扩大演示范围、增强可演示性与一致性**为准。分 P 级标注，P3 默认不做仅记录。

| 编号 | 建议 | 理由（业界范式） | 优先级 | 归宿 |
|------|------|------------------|--------|------|
| S-01 | **触控目标最小尺寸**写入 design-tokens（建议主按钮高 ≥ 64dp，可点区 ≥ 48dp） | 车载大目标降低误触 | P1 | REQ-026 + design-tokens |
| S-02 | **行车时整页降噪态**约定（不仅按钮置灰，输入类一律禁用 + 语音占位统一样式） | AAOS 行车锁定一致性 | P1 | REQ-027 |
| S-03 | **焦点/可达性**：键盘/旋钮焦点顺序与可见焦点环（车机常有物理旋钮） | AAOS 可达性 | P2 | REQ-028 |
| S-04 | **toast 显示时长与并发规则**（同一时刻只显示最高优先级一条，时长固定） | 降噪一致性，呼应拦截优先级 | P1 | REQ-029 |
| S-05 | **空/加载/错误三态统一规范**（骨架屏占位与空状态文案已有，补错误重试态） | 业界三态完备 | P2 | REQ-030 |
| S-06 | **数量步进控件无障碍**（达 1 或 5 边界时 +/- 置灰而非报错） | 已隐含，显式化 | P2 | REQ-031 |
| S-07 | **Demo Bar 状态持久可见 + 当前组合提示语**（如「未登录·驻车·在线」） | 状态可见性，便于评审 | P2 | REQ-032 |
| S-08 | **会员幂等**：已 ACTIVE 再次进入开通流应被引导到「查看订单」而非重复下单 | 防重复，已部分覆盖，显式化 | P1 | REQ-033 |
| S-09 | **语音占位为"假能力"**：点击仅提示 COPY-004，不接真实语音 | 控制演示范围 | P0（边界） | 已在 REQ-003，补强 |
| S-10 | **多语言/明暗**：明确排除（已在不做范围） | 控制范围 | — | 维持排除 |

> 不采纳：真实支付、真实语音、推荐算法、评价/客服、优惠券体系——均超出 Demo 范围，列入排除项维持。

## 4. 结论

本需求在「行车安全门禁、单屏免滚动、扫码支付、服务占位」上已与业界范式对齐。补充集中在**量化触控/降噪 token、三态规范、幂等与状态可见性**，均为低成本一致性增强，已转化为 REQ-026..033。Phase 2 设计据此推进。
